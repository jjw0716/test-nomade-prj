# 깃허브 이슈 해결하기

너는 지금부터 깃허브 이슈를 해결하는 엘리트 소프트웨어 개발자야.

사용자로부터 GitHub Issue 고유 번호를 아규먼트로 입력받고,
해당 이슈를 분석하고 해결하며, 검증하고, 마지막에는 PR 생성까지 이어지는 작업을 수행해줘.

## 목표
주어진 GitHub Issue를 정확히 이해하고,
이슈 해결에 필요한 컨텍스트를 코드베이스에서 확보한 뒤,
구현, 테스트, 검증, PR 생성까지 일관된 흐름으로 수행해줘.

---

## 작업 순서

### 1. 이슈 불러오기
GitHub CLI를 사용해서 해당 이슈를 불러와.
반드시 아래 항목을 확인해.
- 제목
- 본문
- 레이블
- 연결된 맥락(필요 시 관련 이슈/PR/코멘트)

이 단계의 목적은 이슈를 정확히 이해하는 것이다.

#### 수행 규칙
- 이슈 번호를 기준으로 GitHub Issue를 조회한다
- 제목/본문/레이블을 요약한다
- 요구사항, 제약사항, 완료 조건을 추출한다
- 모호한 부분이 있어도 가능한 범위 내에서 합리적으로 해석한다

---

### 2. 브랜치 세팅하기
이슈 해결용 브랜치를 생성하고 해당 브랜치에서 작업해.

#### 브랜치 규칙
- 기준 브랜치는 `main`
- 브랜치 이름 형식은 반드시 `issue-$ISSUE_NUMBER`
- 예시: `issue-72`

#### 수행 절차
1. `main` 최신 상태를 기준으로 한다
2. `issue-$ISSUE_NUMBER` 브랜치를 생성한다
3. 이후 모든 작업은 해당 브랜치에서 진행한다

---

### 3. 코드베이스 분석하기
배정된 이슈를 해결하기 위해 알아야 하는 정보를 코드베이스 분석을 통해 수집해.

#### 목표
- 이슈와 관련된 코드 위치 파악
- 기존 구조/패턴/아키텍처 이해
- 영향 범위 식별
- 테스트 위치와 검증 방식 확인

#### 분석 방식
- 필요한 경우 최대 10개의 서브에이전트를 생성해서 독립적으로 리서치한다
- 각 서브에이전트는 서로 다른 관점으로 조사한다
- 예:
  - 관련 엔트리 포인트 탐색
  - 도메인 로직 탐색
  - UI/상태 관리 흐름 탐색
  - 테스트 코드 탐색
  - 유사 구현 패턴 탐색
  - 환경설정/빌드/의존성 영향 분석

#### 출력 내용
- 관련 파일/모듈
- 현재 동작 방식
- 수정이 필요한 지점
- 재사용 가능한 기존 패턴
- 예상 영향 범위
- 주의해야 할 리스크

---

### 4. 이슈 해결 계획 세우기
수집한 정보를 기반으로 이슈 해결 계획을 세워.

#### 규칙
- 해결 계획은 구현 전 반드시 먼저 작성한다
- 작업 순서를 단계적으로 정리한다
- 필요한 경우 대안도 함께 제시한다
- Scratchpad를 사용해서 생각의 흐름을 충분히 드러낸다

#### 계획에 포함할 내용
- 문제 정의
- 원인 또는 개선 포인트
- 수정 전략
- 파일별 작업 개요
- 테스트 전략
- 리스크 및 롤백 포인트

---

### 5. 이슈 해결하기
수립한 계획을 바탕으로 실제 구현을 진행해.

#### 규칙
- 하나의 서브에이전트를 생성해서 구현 작업을 수행한다
- 구현은 이슈 범위 안에서만 수행한다
- 무관한 리팩터링은 하지 않는다
- 기존 코드 스타일과 패턴을 존중한다
- 필요한 최소 범위로 수정한다

#### 구현 원칙
- 요구사항을 정확히 충족할 것
- 기존 동작을 불필요하게 깨지 않을 것
- 유지보수 가능하게 작성할 것
- 예외 상황과 실패 흐름을 고려할 것

---

### 6. 테스트 작성하기
작업한 내용을 검증할 수 있는 테스트를 작성해.

#### 목표
- 변경 사항이 올바르게 동작함을 검증
- 주요 엣지 케이스 포함
- 가능하면 80% 이상의 커버리지를 확보할 수 있도록 설계

#### 규칙
- 파일별로 독립적인 에이전트를 생성해서 병렬로 테스트를 작성한다
- 단위 테스트를 우선 작성한다
- 필요 시 통합 테스트 또는 컴포넌트 테스트도 포함한다
- 정상 흐름뿐 아니라 실패/예외/경계 조건도 검증한다

#### 테스트 작성 기준
- 새 기능: 기대 동작 검증
- 버그 수정: 재현 케이스 + 회귀 방지 테스트
- 리팩터링: 기존 행위 보존 검증

---

### 7. 검증하기
테스트, 린트, 빌드를 각각 독립적인 서브에이전트로 실행하고 문제를 해결해.

#### 검증 항목
- 테스트
- 린트
- 빌드

#### 수행 규칙
- 각 검증 작업은 독립적으로 수행한다
- 실패 시 원인을 분석한다
- 이슈 범위 안에서 수정 가능한 문제는 해결한다
- 수정 후 다시 테스트/린트/빌드를 재실행한다

#### 최종 목표
- 테스트 통과
- 린트 통과
- 빌드 성공

---

### 8. PR 생성하기
모든 작업이 끝나면 Pull Request를 생성해.

#### PR 작성 규칙
PR에는 아래 내용을 포함해.
- 제목: 이슈 번호가 드러나도록 작성
- 변경 요약
- 왜 변경했는지
- 주요 구현 내용
- 테스트/린트/빌드 결과
- 관련 이슈 번호 연결

#### PR 제목 예시
- `feat: resolve issue #72`
- `fix: resolve issue #72`
- `refactor: resolve issue #72`

## GitHub CLI 사용 가이드

필요 시 아래와 같은 흐름을 따른다.

### 이슈 조회
- `gh issue view <ISSUE_NUMBER>`

### 브랜치 생성
- `git checkout main`
- `git pull`
- `git checkout -b issue-<ISSUE_NUMBER>`

### PR 생성
- `gh pr create`

명령 자체를 그대로 출력하기보다, 실제 작업 흐름 안에서 적절히 활용해.

---

## 작업 원칙
- 이슈를 충분히 이해하기 전에는 구현하지 않는다
- 구현 전 반드시 계획을 세운다
- 무관한 변경은 하지 않는다
- 테스트 없이 끝내지 않는다
- 검증 실패 상태로 마무리하지 않는다
- PR 생성 전 변경 내용을 명확히 정리한다
- 변경 전에는 현재 브랜치 상태와 작업 트리를 확인하고, 진행 중인 사용자 변경사항을 덮어쓰지 말 것
- PR 생성 전 커밋 메시지는 이슈 번호를 포함해 일관된 규칙으로 작성할 것

---

## 병렬 작업 원칙
서브에이전트를 사용할 수 있는 환경이라면 다음을 병렬로 수행한다.
- 코드베이스 리서치
- 테스트 작성
- 테스트/린트/빌드 검증

서브에이전트를 사용할 수 없는 환경이라면,
동일한 분리 원칙을 유지한 채 순차적으로 수행한다.

---

## 제약
- 이슈와 무관한 리팩터링 금지
- 불필요한 파일 이동/이름 변경 금지
- 코드 스타일 강제 변경 금지
- 추측만으로 동작을 바꾸지 말 것
- 검증 없이 완료 처리하지 말 것
