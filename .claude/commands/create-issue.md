# 깃허브 이슈 생성하기

너는 지금부터 개발 요청을 분석해 명확하고 실행 가능한 GitHub Issue를 작성하는 전문가야.

**생성할 이슈 내용:**
$ARGUMENTS

---

## 목표

입력받은 내용을 분석해 GitHub Issue를 작성하고 생성한다.
이슈는 담당자가 추가 질문 없이 바로 착수할 수 있을 만큼 명확해야 한다.

---

## 작업 순서

### 1. 입력 분석

$ARGUMENTS를 읽고 아래 항목을 파악한다.

- **요청 유형 판단**
  - `feat` — 새로운 기능 추가
  - `fix` — 버그 수정
  - `refactor` — 동작 변경 없는 코드 개선
  - `chore` — 설정, 의존성, 빌드 관련
  - `docs` — 문서화

- **핵심 목적** — 이 이슈가 왜 필요한가
- **영향 범위** — 어떤 파일/기능/컴포넌트가 바뀌는가
- **완료 조건** — 언제 "완료"라고 볼 수 있는가

### 2. 코드베이스 참조 (선택)

입력 내용이 기존 코드와 연관된 경우, 관련 파일·컴포넌트·함수를 파악해
이슈 본문에 구체적인 파일 경로나 함수명을 포함한다.
코드베이스와 무관한 요청(문서, 설정 등)은 이 단계를 건너뛴다.

### 3. 이슈 초안 작성

아래 형식으로 이슈를 작성한다.

**제목 규칙**
- 유형 prefix를 붙인다: `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`
- 50자 이내, 행위 동사로 시작한다
- 예: `feat: 도시 상세 페이지 추가`, `fix: 필터 초기화 시 전체 목록 미표시 버그`

**본문 구조**
```
## 배경
왜 이 작업이 필요한지 1~3문장으로 설명한다.

## 작업 내용
- [ ] 구체적인 구현 항목 1
- [ ] 구체적인 구현 항목 2
- [ ] ...

## 완료 조건
- [ ] 완료 여부를 판단할 수 있는 기준 1
- [ ] 완료 여부를 판단할 수 있는 기준 2

## 참고
- 관련 파일: (있는 경우만)
- 관련 이슈/PR: (있는 경우만)
```

**레이블 선택 규칙**
| 유형 | 레이블 |
|---|---|
| feat | `enhancement` |
| fix | `bug` |
| refactor | `refactor` |
| chore | `chore` |
| docs | `documentation` |

### 4. 이슈 생성

초안을 확정한 뒤 GitHub CLI로 이슈를 생성한다.

```
gh issue create \
  --title "<제목>" \
  --body "<본문>" \
  --label "<레이블>"
```

생성 후 반환된 이슈 URL을 출력한다.

---

## 출력 형식

이슈 생성 전 반드시 아래 형식으로 초안을 먼저 출력하고, 이후 생성을 실행한다.

```
[이슈 초안]

제목: <제목>
레이블: <레이블>

본문:
---
<본문 전체>
---

[생성 결과]
URL: <생성된 이슈 URL>
```

---

## 작성 원칙

- 입력이 짧거나 모호해도 코드베이스 맥락을 참고해 구체적으로 보완한다.
- "작업 내용"은 담당자가 체크리스트로 바로 사용할 수 있게 작성한다.
- 추측으로 범위를 과도하게 확장하지 않는다. 불확실한 내용은 "참고" 섹션에 메모로 남긴다.
- 레이블이 저장소에 없을 수 있으므로 `--label` 플래그 실패 시 레이블 없이 재시도한다.
