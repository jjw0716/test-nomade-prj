# 문제 분해 커맨드

너는 지금부터 복잡한 문제를 독립적이고 실행 가능한 태스크로 분해하는 시스템 설계 전문가야.
아래 문제를 분석하고, 각 요소를 고려해 태스크 목록을 작성해줘.

**해결해야 할 문제:**
$ARGUMENTS

---

## 분해 원칙

### 독립성
- 각 태스크는 다른 태스크의 내부 구현에 의존하지 않아야 한다.
- 병렬로 시작할 수 있는 태스크는 명시적으로 구분한다.

### 단일 책임
- 태스크 하나는 하나의 목표만 가진다.
- 조사, 설계, 구현, 검증이 섞이지 않도록 분리한다.

### 실행 가능 크기
- 한 번의 집중된 작업으로 완료 가능한 단위로 유지한다.
- 완료 기준이 불명확하거나 범위가 너무 크면 반드시 세분화한다.

### 의존성 명시
- 선행 작업이 필요한 경우에만 의존성을 명시한다.
- 의존성이 없으면 반드시 "없음"으로 표기한다.

---

## 분해 실행 순서

1. **문제 파악** — 핵심 목적, 변경 영향 범위, 관심사(UI / 로직 / 데이터 등)를 식별한다.
2. **후보 도출** — 관심사와 영향 범위를 기준으로 태스크 후보를 나열한다.
3. **크기 검증** — 각 태스크가 한 컨텍스트 내에서 완료 가능한지 확인하고, 너무 크면 세분화한다.
4. **중복 통합** — 책임이 겹치거나 동일 파일/모듈을 건드리는 태스크는 통합한다.
5. **의존성 연결** — 순서가 중요한 경우에만 의존성을 연결한다.
6. **병렬 그룹 구분** — 의존성 없이 동시에 시작 가능한 태스크를 묶어 표시한다.

---

## 출력 형식

아래 형식을 반드시 준수한다.

```
[문제 분석 요약]
- 핵심 목적:
- 변경 영향 범위:
- 주요 관심사 (예: UI / 비즈니스 로직 / 데이터 / 인프라):

[병렬 실행 가능 그룹]
- 그룹 A (동시 시작 가능): #태스크번호, #태스크번호
- 그룹 B (그룹 A 완료 후 시작): #태스크번호
- ...

[태스크 목록]

#1. [태스크 제목]
- 분류: (프런트엔드 / 백엔드 / 풀스택 / 인프라 / 기타)
- 목적: 이 태스크로 달성하려는 것
- 범위: 수정/생성할 파일 또는 모듈
- 산출물: 완료 시 존재해야 하는 결과물
- 완료 기준: 완료 여부를 판단할 수 있는 구체적인 조건
- 의존성: 없음 또는 #태스크번호
- 병렬 가능: 예 / 아니오

#2. [태스크 제목]
...
```

---

## 출력 규칙

- 태스크 제목만 봐도 해야 할 일을 알 수 있도록 구체적으로 작성한다.
- 추상적인 표현("개선", "최적화", "정리") 대신 행위 동사로 시작한다 ("제거", "추가", "분리", "교체").
- 태스크가 5개를 초과하면 Epic(상위 묶음) → Task(하위 실행 단위) 2단계 구조로 출력한다.
- 모호한 요청이라도 가능한 범위 내에서 실무적으로 해석하고, 가정한 내용은 문제 분석 요약에 명시한다.
